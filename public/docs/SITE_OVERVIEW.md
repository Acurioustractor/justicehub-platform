## What is JusticeHub?

JusticeHub is a digital platform serving **all young people** in Australia's youth justice system. The platform is built under **Indigenous leadership** because Aboriginal and Torres Strait Islander young people are 17 times more likely to be in detentionâ€”making this the sector's most urgent crisis.

**Simple version:** We're building a website where communities can share what works in youth justice, get paid fairly for their knowledge, and eventually own the whole platform themselves.

**Why this matters:** Right now, good programs stay hidden in the communities where they started. JusticeHub makes them visible to everyone who needs themâ€”Indigenous, CALD, remote, disability, and LGBTQIA+ communities all benefit.

## Simple Language Glossary

**ACCO** = Aboriginal Community Controlled Organisation (Indigenous-run, Indigenous-governed)

**CALD** = Culturally and Linguistically Diverse (people from non-English-speaking backgrounds)

**Data Sovereignty** = Communities control their own dataâ€”they decide what gets shared and who sees it

**Economic Self-Determination** = Communities generate their own income instead of depending on grants

**Indigenous Leadership** = Aboriginal and Torres Strait Islander people lead the design and governance

**Philanthropic Dependency** = Always needing grants to survive (we're trying to break this cycle)

**Revenue Model** = How the platform makes money (memberships, training, consulting)

**Sector-Wide Platform** = Serves ALL young people, not just one group

**Youth Justice** = The system that deals with young people who've had trouble with the law

### Database Architecture

**Core Tables:**
- `profiles` - User profiles (young people, practitioners, organizations)
- `programs` - Youth justice programs with full metadata
- `stories` - Community stories and testimonials
- `articles` - Blog posts and thought leadership
- `organizations` - Community organizations and service providers
- `international_programs` - Global best practices
- `empathy_ledger_*` - Relationship tracking between entities

**Key Design Decisions:**
1. **RLS Everywhere** - Row Level Security on all tables for data protection
2. **Soft Deletes** - Nothing is truly deleted, just flagged as inactive
3. **Audit Trails** - Created/updated timestamps and user tracking
4. **Relationship Mapping** - "Empathy Ledger" connects profiles, programs, organizations
5. **Consent Tracking** - Every story/profile tracks consent status

### Technical Challenges & Solutions

**Challenge 1: Complex Relationships**
- **Problem:** Many-to-many relationships between profiles, programs, organizations
- **Solution:** "Empathy Ledger" auto-linking system with manual override
- **Status:** Implemented, ongoing refinement

**Challenge 2: Data Sovereignty at Scale**
- **Problem:** Communities need control but platform needs to function
- **Solution:** Layered consent system with clear ownership boundaries
- **Status:** Implemented at database level, UI workflows in progress

**Challenge 3: Mobile Accessibility**
- **Problem:** Many community members access via mobile only
- **Solution:** Mobile-first design, considering offline PWA
- **Status:** Design responsive, offline features planned for Year 2

**Challenge 4: Performance with Growth**
- **Problem:** Database will grow quickly with 6+ communities contributing
- **Solution:** PostgreSQL indexing, query optimization, caching strategies
- **Status:** Foundation solid, monitoring as we scale

### How to Contribute (For Developers)

**Getting Started:**
1. Clone the repo from GitHub
2. Install dependencies: `npm install`
3. Set up `.env.local` with Supabase credentials
4. Run development server: `npm run dev`
5. Access at `http://localhost:3000`

**Key Directories:**
- `/src/app` - Next.js pages and routing
- `/src/components` - Reusable React components
- `/src/lib` - Utility functions and database clients
- `/supabase/migrations` - Database schema changes
- `/public/docs` - Wiki markdown content

**Making Changes:**
1. Create feature branch from `main`
2. Make changes with clear commit messages
3. Test locally
4. Push and create pull request
5. Request review (if team is established)
