## JusticeHub Unified Search Architecture & Cross-Site Extension

**What exists:**
- Unified Search Layer (/src/lib/search/unified-search.ts) — single entry point consolidating multiple data sources via promise.all parallel queries
- Intent Detection Engine (/src/lib/search/intent-detector.ts) — patterns for find_program, find_person, find_organization, find_media, find_research, general intents + state detection + entity type classification
- Internal Search Provider (/src/lib/search/providers/internal.ts) — queries 4 Supabase tables: alma_interventions, services, organizations, public_profiles via sanitized ILIKE with relevance scoring
- Empathy Ledger Provider (/src/lib/search/providers/empathy-ledger.ts) — fetches media (images/videos/stories) from EL Content Hub API with cultural tag filtering
- Search API Route (/src/app/api/search/route.ts) — GET endpoint with q, type, state, limit params; returns SearchResponse with results + counts + timing
- React Hook (useJusticeSearch.ts) — state management for search results, facets, intent, pagination, filters; calls /api/intelligence/search
- UI Components: AISearch.tsx (debounced AI-enhanced input with suggestions) + QuickSearch.tsx (quick results dropdown with result type icons/badges)
- Search Result Types: intervention, service, person, organization, media, story, research, news (defined in types.ts)
- Faceting: Results grouped by type, state, and source with per-facet counts
- Deduplication & Scoring: Results merged across sources, deduplicated by type:id, sorted by relevance score (0-1 scale)
- Metadata model: state, organizationId, organizationName, category, tags, imageUrl, elderApproved, culturalSensitivity, createdAt
- Available DB tables: alma_interventions, services, organizations, public_profiles + empathy_ledger media + alma_outcomes, program_outcomes, alma_quality_metrics, story_metrics, alma_funding_data, alma_government_programs, civic_metric_snapshots, children_commissioner_reports, and 60+ other tables

**Reusable for the vision:**
- Intent Detection Pattern — INTENT_PRIORITY + INTENT_PATTERNS regex arrays are extensible: add new intents (find_metric, find_reporting, find_site) with keyword patterns without changing core logic
- Provider Pattern — SearchProvider interface is minimal (name, search(), isAvailable()); three new providers for metrics, reporting, cross-site discovery can plug into createJusticeSearch() without modifying existing code
- Parallel Search Architecture — Promise.allSettled() in unified-search.ts lines 92-106 already handles multiple sources with graceful fallback; can scale to 6+ providers with same pattern
- Relevance Scoring — calculateScore() in internal.ts (250-286) uses title match + description match + word boundary bonus + optional boosters (elder_approved); same pattern applies to program outcomes + metrics + people across sites
- Metadata Extensibility — SearchResultMetadata is JSONB with state, organizationId, category, tags, elderApproved, culturalSensitivity, createdAt; can add siteId, reportingYear, metricType, benchmarkGroup without schema change
- Faceting Architecture — buildFacets() (lines 224-256 unified-search.ts) counts by byType, byState, bySource; can add byOrganization, bySite, byMetricCategory for cross-site aggregation
- API Response Shape — UnifiedSearchResponse already includes pagination.total, facets.total, timing.sources, suggestions, warnings; client can display 'Results found in 3 sites' from facets.bySource
- URL Routing — getResultUrl() in QuickSearch (lines 312-333) has 8 cases; can be extended to /metrics/{id}, /reports/{id}, /sites/{siteId}/programs for new result types
- Context-Aware Filtering — JusticeSearchContext accepts organizationId, state, entityTypes, elderApprovedOnly, culturalTags; can extend with siteId, reportingYear, metricType for JR-specific scoping
- UI Component Reuse — QuickSearch + AISearch components are agnostic to result types; can render metrics + reports without UI changes if result types + TYPE_ICONS/TYPE_LABELS maps are extended

**Gaps:**
- NO cross-site search provider — currently only query 1 JusticeHub instance; need JRSiteProvider to discover/aggregate results from N JR sites (NSW, QLD, etc.) via federation API or direct DB queries
- NO metrics/reporting provider — tables exist (alma_outcomes, program_outcomes, alma_quality_metrics, civic_metric_snapshots) but not in search layer; need MetricsProvider to index outcomes + KPIs + benchmark data
- NO site/jurisdiction discovery — no sitesIndex or JRSiteRegistry table to list all 8 JR sites + their URLs/APIs/regions; can't route cross-site queries without site manifest
- NO reporting/dashboard queries — children_commissioner_reports, state_tenders, alma_government_programs tables exist but unreachable from search; need ReportingProvider to surface policy documents + government data
- NO site-specific intent patterns — intent-detector.ts has no patterns for 'show metrics for NSW', 'report from [site]', 'benchmark programs across sites'; need site-aware intent detection
- NO federated search API — unified-search.ts assumes all providers query local Supabase; federated architecture (HTTP calls to remote JR sites) not implemented; would need provider factory + timeouts + result merging
- NO result linkage across sites — results from Site A cannot link to related results in Site B; need cross-site relationship table (story links program in different site) or URL-based discovery
- NO search result cache/history — every query hits DB; no materialized search index, no saved queries, no trending searches across JR network to surface discovery
- NO accessibility for site-specific depth — search for 'NSW mentoring programs' returns 10 results from alma_interventions; can't scope to JusticeHub.nsw, JR_NSW.com, or NSW-specific database without site awareness
- NO multi-language/cultural variants — search doesn't account for Aboriginal community terminology or site-specific cultural frameworks; empathy-ledger has culturalTags but no mapping to JR site cultural contexts

_Currently, JusticeHub search is single-site (one Supabase instance) with 2 providers (internal Supabase + Empathy Ledger media). To extend to cross-site JR network search:

1. **Data Architecture Change**: Instead of 4 searchable tables, need 3 new providers:
   - JRNetworkProvider: discovers sites, routes queries to /api/search endpoints on remote JR instances
   - MetricsProvider: exposes alma_outcomes + program_outcomes + alma_quality_metrics in search results
   - ReportingProvider: surfaces children_commissioner_reports + policy documents + government programs

2. **Intent Detection Expansion**: Add patterns for 'metrics', 'reporting', 'benchmark', 'across sites' to detect cross-site queries vs local queries

3. **Site Manifest**: Create jr_sites table listing { id, name, state, region, api_url, logo, primary_contact } — enables provider to discover + call remote endpoints

4. **Result Type Expansion**: Add metric, report, benchmark, funding_opportunity to SearchResultType enum; update TYPE_ICONS/TYPE_LABELS in UI

5. **Scope API**: Extend JusticeSearchContext with { siteIds?: string[], reportingYear?: number, metricType?: string } to filter cross-site results

6. **Faceting**: buildFacets() already groups bySource; when bySource includes 'jr-nsw', 'jr-qld', etc., UI can show 'Found in 3 states'

7. **Federated Timeouts**: Promise.allSettled() already handles failed sources gracefully; remote provider timeouts won't block results

8. **Result Deduplication**: Extend dedupeAndSort() to handle 'program_outcome' as distinct type from 'program' so same intervention + its outcomes both surface

See also: /src/lib/search/types.ts (SearchProvider interface), /src/lib/search/unified-search.ts (createJusticeSearch factory), /src/lib/search/intent-detector.ts (extensible pattern matching)_

## JusticeHub community profile ownership lifecycle: claim → own → edit, spanning public action-profiles to internal org hubs, with emerging JR site-to-site linking.

**What exists:**
- Anchor communities system: 4 founding org profiles (Oonchiumpa, Palm Island CC, BG Fit, Minjerribah Moorgumpin Elders) in /communities/[slug], serving as the foundation for community-controlled profile ownership
- Claim form flow: /communities/[slug]/claim (ClaimProfile.tsx) + POST /api/communities/claim → upserts contact into GHL with organization-scoped tags + human review workflow
- Organization claims table (organization_claims): pending/verified/rejected/revoked status, contact details, ABN, consent_level, permitted_uses, story_interest, reversibility_until window (14 days)
- Organization hub interior: /hub/[org-slug]/profile (DashboardView.tsx + tabs: identity, programs, people, proof, public) for claimed orgs to edit their workspace after verification
- Profile content model (from org-profile-spec.md): identity (name, place, indigenous badge), programs (alma_interventions, non-ai-generated), outcomes (alma_evidence, consent-gated), cost picture (justice_funding vs ROGS detention), funding history, stories (EL v2 API), network relationships
- Justice Reinvestment site enrichment: curated sites.json (34 placed sites) + optional org-connections.json (ABN, board links, relatedSites, revenue, charity status) feeding the JR network map + sidebar detail index
- Admin profile creation from JR connection: POST /api/admin/organizations/create-from-connection (HYBRID dedup: ABN-primary, slug-fallback, tagging with jr-site:<id> for no-ABN rows)
- Org profile workbench: 5-tab interface showing identity (CivicScope sync + JH editable), programs/services, people (public team + CivicScope board roles), proof (stories + empathy ledger consent ledger), public outputs (org profile URL + funding workspace + funder proof page)
- Consent tiers in proof tab: Public Knowledge Commons (published stories/media for case studies), Community Controlled (draft stories held by org), Strictly Private (EL-held, not mirrored into JH)
- Verification ladder: unclaimed (public records) → org-confirmed (org reviewed/approved) → community-verified (outcomes confirmed with community evidence)
- Organization membership layer: organization_members table with roles (member, admin, staff, counselor, manager, owner, editor, viewer), status (active/pending), scoped RLS policies (is_org_member function maps owner/editor/viewer levels)
- Invite token flow in organization_claims: verified claim → invite_token + invite_expires_at → claimant accepts → owner membership (infrastructure in 20260611000001 migration, unused by front-end)
- Claim submission service (submitOrganizationClaim): validates role/email/consent, finds/creates org by ID/ABN/gs_entity_id, inserts organization_members.primary_organization_id, queues org_claim_review task, sends confirmation email
- Public organization profile (/organizations/[slug]): reads programs, people, articles, photos, videos, media_items, member count, GrantScope dossier (funding, contracts, relationships, board roles, interventions)
- Slug stability: suggestProfileSlug helper strips legal suffixes (inc, ltd, pty), lowers, kebab-cases; uniqueOrgSlug adds numeric suffix on collision; jrSiteTag creates namespaced tag 'jr-site:<slugified-match-name>' for tracking curated-to-profile lineage

**Reusable for the vision:**
- Claim → own → edit lifecycle blueprint: unauthenticated claim form (GHL contact capture) → human review (agent task queue) → approval (invite_token flow) → member activation → workspace access. This pattern is foundational for JR network co-ownership.
- HYBRID dedup strategy for site-to-org linking: primary on ABN (the hybrid account key), fallback to slug (soft dedup), tag-based fallback for no-ABN sites. Avoids mis-linking different orgs while reusing auspice accounts.
- Namespaced tag architecture (jr-site:<id>): enables deterministic reverse-lookup from curated JR sites to profile accounts without colliding with other tag vocabularies. Decouples profile slug from URL slug.
- 5-tab workbench model (identity, programs, people, proof, public) with visibility framing: each tab has a 'what to publish next' guide + completeness checklist. Proof tab mirrors EL consent tiers exactly.
- Consent-tier tripartite split (Public Knowledge Commons / Community Controlled / Strictly Private): already SQL-enforced in organization_claims, mirrors Empathy Ledger vocabulary, ready to extend to per-story/per-program granularity.
- Verification ladder (unclaimed → org-confirmed → community-verified): honest three-tier signal, displayed on profiles. Moves verification from binary (verified/unverified) to process-stage (public records → org review → evidence confirmation).
- Profile anchor data model: fixed ANCHOR_COMMUNITIES list with ILIKE pattern for DB determinism + fallback identity (for cases where org is not yet in database). Allows rendering profile even when DB lookup fails.
- Enrichment indexing pattern: load curated layer once on server (sites.json + org-connections.json), build indexes (match_name → enrichment), pass as serialized record to client. No per-click fetches.
- Organization membership role system: 8-role hierarchy (member → staff/counselor/manager/admin/editor/owner/viewer) with RLS function is_org_member(org, min_role) for scoped policies. Ready for fine-grained dashboard access control.
- Org profile workspace data model: fetches programs (programs_catalog_v), people (organizations_profiles + nested public_profiles), articles, photos, videos, media_items separately. Each section safe-lists (catches errors), allowing partial rendering when data sources fail.

**Gaps:**
- Site-to-site linking flow is HALF-BUILT: org-connections.json carries relatedSites array, but no front-end surface exists to navigate from one profile to related orgs. The JR map sidebar shows detail (programs, funding, claim status) but does not render a 'related sites' card or 'linked network' view.
- Co-story capturing is NOT IMPLEMENTED: the consent tiers (Public Knowledge Commons / Community Controlled / Strictly Private) are defined in claim-service.ts and org-profile-spec.md, but no UI exists to assign consent tiers to individual stories or capture consent-per-story. Stories are read-only from Empathy Ledger; no local consent metadata is stored in organization_claims or elsewhere.
- Invite-token acceptance flow is INFRASTRUCTURE-ONLY: organization_claims.invite_token + invite_expires_at columns exist (migration 20260611), but no route exists to accept the token, activate membership, or email the invite link. claim-service.ts creates the claim but does NOT generate or email the token.
- No profile publishing gate: the workbench allows org admins to edit identity/programs/people/proof, but there is NO 'publish' action or visibility toggle. All changes are live-write; the org profile immediately reflects edits. The spec calls for 'JusticeHub staff can stage, never publish' but the code is all-or-nothing.
- No cost-picture block (program cost vs ROGS detention benchmark): org-profile-spec.md lists this as step 4 in the build order, but no UI exists. justice_funding records are fetched in dossier (GrantScope) but not rendered in the cost context.
- No spoke-circuit offline form: the spec calls for a capture flow for the Australia trip (offline-tolerant form staging), but no such form exists. The claim form is web-only.
- RLS policies on organization_members are NASCENT: admin_all policy on organization_claims exists, but organization_members policies (org_members_select_roster, org_members_owner_insert) reference is_org_member but no test coverage or admin API routes exercise them yet.
- No verification-ladder display in public profile: the profile shows verification_status (a string), but the three-tier ladder (unclaimed → org-confirmed → community-verified) is not rendered, and the status vocabulary in the DB does not match the spec ladder names.
- Empathy Ledger integration is READ-ONLY: anchor.empathyLedgerProjectId exists, but there is no route to link/unlink an EL project to an org, no consent-sync from EL, and no UI to manage the EL relationship in the workbench.
- No audit trail for profile edits: organization_claims have an audit_events infrastructure (admin create-from-connection logs), but the workbench tabs (editing identity/programs/people) have no audit logging or change history.
- ALMA interventions are read-only: the spec marks alma_interventions as the Programs section source, with the ai_generated filter enforced, but there is no UI to create, edit, or link new interventions. All delivery records come from external sources.
- No network visualization or co-story discovery: the JR map shows sites on a map and curated connections as sidebar detail, but there is no 'explore related sites', 'see co-stories', or 'map my network' view for an org.
- Profile claiming does NOT detect existing org records: the claim form does not pre-fill org name from a lookup, does not offer auto-suggest, and does not check if the claimant is already listed in organization_members of that org.

_The lifecycle exists as a three-stage stack: (1) DISCOVERY: curated JR sites.json + optional org-connections.json enrich a raw database list (alma_interventions + organizations). (2) CAPTURE: claim form (GHL contact capture) + human review (agent_task_queue) + invite-token infrastructure (defined but unused). (3) OWNERSHIP: org hub workbench (5 tabs) with role-based access (is_org_member RLS) and consent tiers (EL mirror). The critical missing piece is the BRIDGE from step 1 to step 3: no front-end flow exists to ACCEPT an invite token, ACTIVATE membership, or EMAIL the invite link. The workbench also lacks a publish gate and audit trail. For co-stories, the infrastructure (consent tiers in claim-service) is defined but the UI to assign consent per-story does not exist. The related_sites array in org-connections.json is loaded but not rendered. The gap is not in the database schema or the helpers layer; it is in the front-end workflows that turn curated data (JR sites) into editable org profiles (the hub) and the user-facing features (co-story consent, network discovery) that make those profiles valuable to communities. The spec (org-profile-spec.md) is also clear on governance: 'The organisation is the editor of record' and 'JusticeHub staff can stage, never publish' — but the code does not enforce these; the workbench is all-or-nothing live._

## JusticeHub Events, Network Membership & Member-to-Member Connection Infrastructure

**What exists:**
- Events table (events) — full schema with event_type, start_date, end_date, location, node_id, registration_url, max_attendees, is_featured, is_public, image_url, video_url, gallery_urls, organization_id, project_id; RLS policies for public read, service role full access
- Events public page (/src/app/events/page.tsx) — lists all public events ordered by start_date with node filtering
- Events admin panel (/src/app/admin/events/page.tsx) — create/edit/manage events; admin/new and admin/[id] routes exist
- Event registrations table (event_registrations) — stores email, full_name, organization, phone, dietary_requirements, accessibility_needs, registration_status (registered/confirmed/cancelled/attended/no_show), notes; unique constraint on (event_id, email); RLS: users see own registrations by email
- JusticeHub Nodes (justicehub_nodes) — geographic backbone table with state/territory/international nodes; seeded with 9 initial nodes (QLD through ACT + Aotearoa)
- Newsletter subscriptions table (newsletter_subscriptions) — for GHL sync; email, full_name, organization, subscription_type (general/steward/researcher/youth), ghl_contact_id, is_active, subscribed_at/unsubscribed_at
- Centre of Excellence key people (coe_key_people) — links public_profiles to roles/expertise with display_order; is_active filter
- Network memberships table (network_memberships) — organization_id (FK), basecamp_id (FK), role ('miner' for ALMA), status (pending/active), contact_name/email/phone, description, focus_areas (text[]), young_people_served (int), approved_at, joined_at; RLS: service role only
- Network join API (POST /api/network/join) — creates/matches org, auto-assigns basecamp by state, creates membership with 'pending' status, syncs to GHL CRM; matches org by ABN then by name+state
- Peer validations table (peer_validations) — org-to-org endorsements; from_org_id, to_org_id, validation_type (endorsement/site_visit/collaboration/referral), content, validator_name/role, is_public, created_at
- Peer validation API (POST/GET /api/network/validate) — allows network orgs to endorse/validate each other; public read filtered by is_public=true
- Matched opportunities API (GET /api/network/matched-opportunities?org_id=xxx) — filters youth & funding opportunities by org state and focus areas
- ALMA Network gathering page (/src/app/network/alma/gathering/page.tsx) — displays national scorecard data (basecamps, members, interventions, funding, validations, stories) from network_memberships.status=active query
- Mentorship tables (mentor_profiles, mentorship_requests, mentorships) — from initial schema but NOT actively integrated into live codebase; mentor_profiles references users, max_mentees, availability_hours, background_check_completed; mentorships has shared_goals, meeting_frequency, progress_updates; mentorship_connections table in empathy_ledger_core.sql with mentor_id, mentee_id, connection_type (one_on_one/group/peer/program_based), connection_strength, meeting_frequency, last_interaction
- Empathy ledger core tables (mentorship_connections, youth_profiles, grassroots_programs) — foundation for tracking mentorship + youth growth journeys; NOT integrated with ALMA network yet

**Reusable for the vision:**
- Events infrastructure is complete and public-facing — can be leveraged for cross-site JR network events without new schema. Just populate events table with cross-regional sessions + link to appropriate node_id
- JusticeHub Nodes backbone (9 state/territory nodes) is seeded and ready — perfect foundation for associating events, mentors, and resources by geographic region
- Network membership flow (join → pending → active) is proven and already syncs to GHL CRM — same pattern can extend to mentor onboarding without schema changes
- Peer validations (org-to-org endorsements) can extend to person-to-person mentorship endorsements by reusing validation_type field + adding mentor-specific types (e.g. 'mentorship_qualified', 'peer_match')
- RLS policies for organization_members (member/admin/editor/viewer roles) provide access control template for mentor-mentee relationships — can repurpose role vocabulary for mentor tiers
- Mentorship_connections table in empathy_ledger_core already has meeting_frequency, last_interaction, connection_strength, shared_goals columns — ready to populate with cross-site mentor pairs without migration
- Matched opportunities API shows pattern for filtering by focus_areas + state — can replicate for 'mentor matching by expertise + location' feature
- Newsletter subscriptions + GHL sync already handles audience segmentation (general/steward/researcher/youth) — can add mentor segments without schema change
- Contact capture (event_registrations.email + contact_name) is already in place — hook into event creation to offer 'be a mentor' opt-in without new UI

**Gaps:**
- Event registrations table has NO mentor-matching or interest capture — registrants provide name/email/organization/accessibility but not expertise/mentorship_interest/mentor_availability; no schema to link register → offer mentoring
- No cross-site mentor directory/discovery surface — mentors are in mentor_profiles table but there's no UI for 'find a mentor by expertise + location' or 'mentors from other JR sites'

- No catch-up/follow-up scheduling after events — event registrations don't track post-event engagement or suggest follow-up connections between attendees from same/different sites
- No peer mentor validation feature — peer_validations exists but is org-only; no person-to-person mentor endorsements or peer recommendation flow
- Mentor availability/calendar surface missing — mentor_profiles has max_mentees + availability_hours but no booking/calendar table to manage 1:1 session times or cross-site catch-ups
