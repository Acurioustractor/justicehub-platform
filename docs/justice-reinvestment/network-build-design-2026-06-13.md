## 1. Framing

A map is not a network. Right now thirty-four sites sit as pins on a Leaflet canvas, each one carrying what we could find for it from the outside. The next move turns the pins into a living room. Each community walks through a door it controls, hangs its own programs and people and proof on the walls, and discovers that the room next door has been working the same problem in a different language. One search box becomes the way a worker in Bourke finds what Doomadgee learned about night patrols, the way a funder in Canberra sees every diversion program against the price of a cell, and the way a gap — a missing evaluation, an unconfirmed place, a program with no evidence beside it — stops being a silence and becomes an invitation. The architecture is three rings: the curated layer the platform holds, the owned layer the community edits, and the connective tissue — search, stories, mentors, catch-ups — that lets one ring read another. Money lives inside that structure, beside detention cost, never floating free. Build it so that the network can read its own body.

## 3. Data Model

### 3.1 Use these EXISTING tables (no new schema)

| Capability | Existing table(s) | Rule |
|---|---|---|
| Programs run by a site | `alma_interventions` (FK `operating_organization_id`) | **MUST** `.neq('verification_status','ai_generated')`. No `location_state` — derive state via `organizations.state` JOIN (already done in `loadJusticeReinvestmentNetwork`). |
| Site → org identity | `organizations` (`state`, `is_indigenous_org`) | No `source` column. |
| Outcomes/evidence | `alma_evidence` (unique `source_url`; **no** `verification_status` column — guard cannot apply) | Consent-gated per item. |
| Cost picture | `justice_funding` (`alma_organization_id` FK, `amount_dollars`) | Partial unique index → use select-then-insert, never `.upsert()`. Every figure needs source provenance. |
| Quantified metrics store | `alma_outcomes`, `program_outcomes`, `alma_quality_metrics`, `civic_metric_snapshots` | These exist and are unindexed by search — Phase 4 surfaces them. Show **confirmed only** (research notes 0/total confirmed across all 34 sites today — render honestly as "not yet confirmed"). |
| Stories / media / people | Empathy Ledger v2 API via `src/lib/empathy-ledger/v2-client.ts`; local `stories`, `storytellers`, `media_items` (array FKs `story_ids[]`, `organization_ids[]`), `public_profiles`, `partner_stories`/`partner_photos` | **NEVER** query EL Supabase directly. Real photos only. |
| Consent / visibility | `src/lib/privacy.ts` (3-tier), `organization_claims.consent_level` | Public Knowledge Commons / Community Controlled / Strictly Private. |
| Claim → own → edit | `organization_claims` (`invite_token`, `invite_expires_at`, `reversibility_until`), `organization_members` (8 roles + `is_org_member()` RLS) | Owning org is editor of record; staff stage, never publish. |
| Site curation layer | `src/data/justice-reinvestment/sites.json` (34 sites), `src/data/justice-reinvestment/org-connections.json` (ABN, charity, `related_sites`, revenue) | Committed JSON, loaded server-side once, passed as serialized index. |
| Events | `events` (`node_id`, `organization_id`, `is_public`), `event_registrations` (unique `event_id`+`email`) | Production-ready, public-facing. |
| Network membership | `network_memberships`, `peer_validations` (org-to-org endorsements) | Proven join flow → GHL sync. |
| Mentors/catch-ups | `mentor_profiles`, `mentorships`, `mentorship_connections` (`connection_type`, `meeting_frequency`, `last_interaction`) | Schema exists, **zero code references** — wire UI/API, no migration needed for the base case. |
| Geographic backbone | `justicehub_nodes` (9 state/territory nodes seeded) | Associate events/mentors/sites by region. |

### 3.2 MINIMUM new schema

Add only what the curated JSON cannot carry across sessions and what links sites to each other and to impact.

**A. `jr_sites` registry table** — promote `sites.json` to a queryable row source so search and per-site pages have stable IDs. Keep the JSON as the seed/source-of-truth import; the table is the runtime index.
```sql
-- migration: 20260614000001_jr_sites_registry.sql
create table jr_sites (
  id uuid primary key default gen_random_uuid(),
  match_name text unique not null,          -- canonical key from sites.json
  slug text unique not null,                -- /communities/justice-reinvestment/[slug]
  display_name text not null,
  state text,
  town text,
  lat double precision,
  lng double precision,
  website text,
  logo_url text,                            -- real logo only, never AI image
  blurb text,
  lead_organization_id uuid references organizations(id),  -- nullable; hybrid auspice
  jr_site_tag text,                         -- 'jr-site:<normalized>' lineage tag
  confirmed_site boolean default false,
  data_quality text check (data_quality in ('rich','moderate','thin')),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

**B. `jr_site_links` table** — site-to-site relationships (today `related_sites` is a one-way string array in JSON, not navigable, not bidirectional).
```sql
create table jr_site_links (
  id uuid primary key default gen_random_uuid(),
  from_site_id uuid references jr_sites(id) on delete cascade,
  to_site_id uuid references jr_sites(id) on delete cascade,
  link_type text check (link_type in ('shared_auspice','peer','co_story','referral','same_model')),
  note text,
  created_by uuid references auth.users(id),
  is_public boolean default true,
  created_at timestamptz default now(),
  unique (from_site_id, to_site_id, link_type)
);
```

**C. `story_related_sites` junction** — stories currently link to programs/orgs only (no place ID). Mirror the existing `story_related_programs` pattern.
```sql
create table story_related_sites (
  story_id uuid not null,                   -- EL v2 id or local stories.id
  jr_site_id uuid references jr_sites(id) on delete cascade,
  relevance_note text,
  display_order int default 0,
  co_story boolean default false,           -- true when authored across 2+ sites
  consent_level text default 'community_controlled',
  primary key (story_id, jr_site_id)
);
```

**D. `jr_site_metrics` store** — confirmed, sourced impact numbers (research shows the rich tables aren't populated for JR sites yet; this is the honest, provenance-bearing landing for confirmed figures, distinct from speculative `alma_*` rows).
```sql
create table jr_site_metrics (
  id uuid primary key default gen_random_uuid(),
  jr_site_id uuid references jr_sites(id) on delete cascade,
  metric_key text not null,                 -- 'young_people_diverted', 'cost_per_case'
  metric_label text not null,
  value numeric,
  unit text,
  reporting_year int,
  source_url text not null,                 -- provenance REQUIRED — no figure without a source
  source_note text,
  confirmed boolean default false,          -- maps to verification ladder
  confirmed_by_org boolean default false,
  created_at timestamptz default now(),
  unique (jr_site_id, metric_key, reporting_year)
);
```

**E. Column adds (no new tables):**
- `event_registrations`: add `mentor_interest boolean default false`, `expertise text`, `availability text` — capture mentor opt-in at registration (closes the documented gap).
- `mentorships`: add `from_jr_site_id uuid`, `to_jr_site_id uuid` references `jr_sites(id)` — cross-site mentor pairs.

That is the whole new surface: **4 tables + 2 column adds.** Everything else reuses what exists.

## 4. Cross-Site Search Design

Extend `createJusticeSearch()` in `src/lib/search/unified-search.ts`. The provider pattern (`SearchProvider` interface: `name`, `search()`, `isAvailable()`) and `Promise.allSettled` fan-out already scale to N providers with graceful fallback. Today: 2 providers (`internalSearchProvider`, `empathyLedgerSearchProvider`). Add three, all in `src/lib/search/providers/`.

### New providers
1. **`jrSiteProvider`** (`providers/jr-sites.ts`) — queries `jr_sites` + `jr_site_links`. Returns `site` results. Lets "Bourke", "Maranguka", "night patrol site" resolve to a site card.
2. **`metricsProvider`** (`providers/metrics.ts`) — queries `jr_site_metrics` (confirmed only) + the existing `alma_outcomes`/`program_outcomes`/`alma_quality_metrics` (guarded). Returns `metric` results.
3. **`reportingProvider`** (`providers/reporting.ts`) — surfaces `children_commissioner_reports`, `alma_government_programs`, `justice_funding` summaries for the government view. Returns `report` / `funding` results.

### Result type expansion
`src/lib/search/types.ts` `SearchResultType` is currently:
`'intervention' | 'service' | 'person' | 'organization' | 'media' | 'story' | 'research' | 'news'`.
Add: `'site' | 'metric' | 'report' | 'funding'`.
Then add matching cases to `getResultUrl()` in `src/components/search/QuickSearch.tsx` (it already switches on all 8 types) and to `TYPE_ICONS` / `TYPE_LABELS`.

### Intent expansion
`src/lib/search/intent-detector.ts` is regex-array extensible (`INTENT_PRIORITY` + `INTENT_PATTERNS`). Add intents without touching core logic:
- `find_site` — "site in", "community at", "[place] JR"
- `find_metric` — "outcomes", "results", "how many diverted", "cost per"
- `find_reporting` — "report", "evaluation", "funded by", "across sites", "benchmark"

### Faceting (already supported)
`buildFacets()` already groups `byType`, `byState`, `bySource`. Cross-site discovery falls out for free: when `bySource` includes the new providers, the UI renders "Found across 3 states / 12 sites." Add `byOrganization` to surface shared-auspice clustering (Just Reinvest NSW covers 5 NSW sites).

### Scope context
Extend `JusticeSearchContext` with `siteIds?: string[]`, `reportingYear?: number`, `metricType?: string` — used by the government view to scope "all diversion programs in QLD."

### Surfacing
- Inline search bar on `/communities/justice-reinvestment` (component reuse: `AISearch.tsx` is result-type agnostic).
- `useJusticeSearch` hook unchanged — it already manages results/facets/intent/pagination.
- Provider timeouts are non-blocking (`Promise.allSettled`), so a slow metrics query never blocks site results.

## 5. Community-Owned Profiles + Site-to-Site Linking + Co-Stories

Built entirely on the existing claim → own → edit lifecycle. The schema and helpers are done; the gaps are front-end workflows.

### Claim → own (close the invite-token gap)
1. Site card on `/communities/justice-reinvestment/[site-slug]` shows a **Claim this profile** button when `organization_claims` has no verified row (reuse `ProfileStatusChip` from `JRSidebar`).
2. Existing `submitOrganizationClaim` (`src/lib/organizations/claim-service.ts`) creates the claim + queues `org_claim_review`. **NEW work:** on admin approval, generate `invite_token`, email it, and add route `/communities/claim/accept/[token]` that activates the `organization_members` owner row. The columns (`invite_token`, `invite_expires_at`) already exist in migration `20260611000001`.
3. Hybrid dedup already handles auspice sharing: ABN-first, slug-fallback, `jr-site:<match_name>` tag (`create-from-connection` route). Reuse as-is.

### Own → edit (add publish gate)
The `/hub/[org-slug]/profile` 5-tab workbench is all-or-nothing live-write today; the spec requires "staff stage, never publish." **NEW work:** add a `published` boolean per section (or a staging shadow), and a Publish action gated on `is_org_member(org, 'owner')`. Add audit logging to the existing `audit_events` table on each edit (the create-from-connection route already writes there — reuse the pattern).

### Site-to-site linking
- New `/hub/[org-slug]/network` tab: a site sees its `jr_site_links` (seeded from `org-connections.json` `related_sites`), proposes new links, accepts/rejects inbound.
- Public render: `/communities/justice-reinvestment/[site-slug]` "Related sites" card (the `JRSidebar` already renders `related_sites` buttons via `onSelectRelated` — promote that to the detail page using `jr_site_links` instead of the JSON array).

### Co-stories
- Stories stay read-from-EL via `v2-client.ts` (never write to EL Supabase). Co-authorship is recorded **locally** in `story_related_sites` (`co_story = true`, two+ site rows for one story).
- `/hub/[org-slug]/profile` proof tab gains per-story consent assignment (reuse `PrivacySelector.tsx`) writing to `story_related_sites.consent_level` — closes the "consent tiers defined but no per-story UI" gap.
- Co-story discovery on `/communities/justice-reinvestment/network`: stories linked to 2+ sites render as connective cards between those sites.

## 6. Impact + Storytelling for Two Audiences

Same data, two lenses. Toggle via `?view=` on the explorer and dedicated routes.

### Community view (default)
**Sees on `/communities/justice-reinvestment/[site-slug]`:**
- Identity + Country/place (real logo from `jr_sites.logo_url` or initials fallback — reuse `JRMap` marker logic).
- Programs (`alma_interventions`, guarded) with type + evidence level.
- Stories + photos (EL v2 API, consent-honored, real photos only) via `story_related_sites`.
- People (`public_profiles` + `community_programs_profiles` roles).
- Funding history (`justice_funding`, largest first) **beside the ROGS detention benchmark** ($1.33M national avg, already in profile page).
- Related sites + co-stories (`jr_site_links`, `story_related_sites`).
- Verification ladder badge (unclaimed → org-confirmed → community-verified).

### Government / funder view
**Sees on `/communities/justice-reinvestment/impact` and `?view=government`:**
- Aggregate confirmed metrics only (`jr_site_metrics WHERE confirmed = true`) — rendered honestly: today that is "0 confirmed, N awaiting confirmation," which is itself the argument.
- Funding grouped by source / department / state (`justice_funding` + `alma_government_programs`).
- Cost-vs-detention roll-up: total program spend across N sites against detention cost, "what scaling to Z would save" (every figure provenance-sourced).
- Diversion-program counts by state (faceted search `byState` + `byType`).
- Downloadable by state, respecting section visibility (no Strictly Private rows ever export).

Both views render the **gap honestly** rather than hiding it — a missing metric is a labelled gap, not a blank.

## 7. Gap Analysis + Capacity + Network Growth + Events/Mentors/Catch-ups

### Gap analysis (computed, not new infra)
A gap is a derivable signal over existing rows. Compute per site:
- No `confirmed_site` → "place to confirm" (already bucketed in `JRNetworkExplorer`).
- No `lead_organization_id` / no ABN in `org-connections.json` → "org unverified."
- `jr_site_metrics` count = 0 → "no confirmed outcomes."
- No `story_related_sites` rows → "no stories yet."
- No `justice_funding` rows → "funding unknown."

Render as gap badges (reuse `VerificationChip`/`ProfileStatusChip` styling) on site cards and an admin gap checklist in `JrLocationsPanel` (already has expandable detail drawer + filters). Each gap links to the action that fills it ("claim this profile," "add a confirmed metric," "link a story").

### Capacity support + network growth
- The `network_memberships` join flow (pending → active → GHL sync) is the growth funnel. A site with gaps gets surfaced opportunities via the existing `matched-opportunities` API (filters by state + focus areas) — extend to suggest peer sites that have already filled the same gap.
- `peer_validations` lets a site endorse another's program/site visit — capacity signal made visible.

### Events / mentors / catch-ups (wire existing tables)
- **Events:** `events` table is production-ready. New `/hub/[org-slug]/events` tab + cross-network events on the public surface, associated by `justicehub_nodes`. No schema change.
- **Mentor capture:** add `mentor_interest`/`expertise`/`availability` to `event_registrations` (§3.2E) so registering for a catch-up offers "be a mentor."
- **Mentor directory + matching:** wire the dormant `mentor_profiles`/`mentorships`/`mentorship_connections` tables to UI + a matching API (replicate `matched-opportunities` pattern: match by expertise + node/state). Cross-site pairs via new `mentorships.from_jr_site_id`/`to_jr_site_id`.
- **Catch-ups:** `mentorship_connections.meeting_frequency` + `last_interaction` already model recurring catch-ups; surface a simple scheduling card. No calendar integration in Phase 4 (see open decision 5).

## 8. Phased Build Plan

Each phase ships standalone. Ordered by value-per-dependency. **Build Phase 2A first** — per-site pages are the unit everything else hangs on, and they ship on data that already exists.

### Phase 2A — Per-site detail pages (FIRST THING TO BUILD)
**Why first:** the map already has all the sidebar data; promoting it to a real page unlocks deep links, search targets, and the canvas for stories/impact. Zero new dependencies.
- **Routes:** `/communities/justice-reinvestment/[site-slug]` (new).
- **Tables:** `jr_sites` registry (migration A) seeded from `sites.json`; reads `alma_interventions` (guarded), `organizations`, `justice_funding`, `org-connections.json`.
- **Files:** new `page.tsx` + reuse `JRSidebar.tsx` section components; extend `src/lib/communities/justice-reinvestment.ts` loaders to resolve by slug; `jr-profile-helpers.ts` slug logic.
- **Acceptance:** every one of the 34 sites resolves at a stable URL; programs render guarded (no `ai_generated`); funding shows beside ROGS benchmark with sources; real-logo-or-initials only; passes DESIGN.md (no em dashes, no `bg-white`/`bg-black`).

### Phase 2B — Cross-site search
**Depends on:** 2A (sites need IDs/URLs to be search targets).
- **Files:** new `providers/jr-sites.ts`; extend `types.ts` (`'site'`), `intent-detector.ts` (`find_site`), `QuickSearch.tsx` (`getResultUrl`/`TYPE_ICONS`). Inline search bar on the JR page.
- **Tables:** `jr_sites`, `jr_site_links`.
- **Acceptance:** searching a place ("Bourke") returns the site card; facets show "found across N states"; existing intervention/story results still work; provider timeout never blocks site results.

### Phase 3 — Community ownership (claim → edit → publish + links + co-stories)
**Depends on:** 2A.
- **Routes:** `/communities/claim/accept/[token]` (new); `/hub/[org-slug]/network` (new tab); publish gate on `/hub/[org-slug]/profile`.
- **Tables:** `organization_claims` (token activation), `organization_members`, `jr_site_links`, `story_related_sites`; `audit_events` for edit logging.
- **Files:** `claim-service.ts` (emit + email token), new accept route, `HubShell.tsx` (network tab), proof tab per-story consent via `PrivacySelector.tsx`.
- **Acceptance:** a verified claim produces an emailed token → acceptance activates an owner membership; an owner edits and explicitly publishes (staff cannot publish); two sites link bidirectionally; a story tagged to 2 sites renders as a co-story with its consent level honored.

### Phase 4 — Impact, metrics, government view
**Depends on:** 2A, 2B.
- **Routes:** `/communities/justice-reinvestment/impact` (new); `?view=government` on the explorer.
- **Tables:** `jr_site_metrics` (migration D); reads `justice_funding`, `alma_government_programs`, `children_commissioner_reports`; new `providers/metrics.ts` + `providers/reporting.ts`.
- **Files:** impact page, government-grouped explorer mode, metric/report result rendering.
- **Acceptance:** every rendered figure has a `source_url`; unconfirmed metrics show as labelled gaps not blanks; government view groups by funding source/state and totals against detention cost; nothing Strictly Private exports.

### Phase 5 — Gap analysis surface
**Depends on:** 2A, 4.
- **Files:** gap-badge computation in JR loaders; admin gap checklist in `JrLocationsPanel.tsx`; gap CTAs on site pages.
- **Tables:** none new (derived over existing).
- **Acceptance:** each site shows accurate gap badges; admin can filter sites by gap type; each gap links to its fill-action.

### Phase 6 — Events, mentors, catch-ups
**Depends on:** 3 (hub tabs).
- **Routes:** `/hub/[org-slug]/events` (new tab); cross-network events on public surface.
- **Tables:** `events`, `event_registrations` (+mentor columns, §3.2E), `mentor_profiles`/`mentorships`/`mentorship_connections` (wire dormant), `justicehub_nodes`.
- **Files:** events tab, mentor directory UI, matching API (clone `matched-opportunities`).
- **Acceptance:** a cross-site event publishes and accepts registrations with mentor opt-in; a mentee finds a cross-site mentor by expertise + state; a catch-up records `last_interaction`.

## 9. Open Decisions for the User (max 6)

1. **Where do public profiles live — `/communities/[slug]` or `/communities/justice-reinvestment/[site-slug]`?** The spec hedges ("could live under `/directory`"). For shared-auspice sites (5 NSW sites under Just Reinvest NSW), do we want one org profile or one page per place? This shapes the slug model and the hybrid-dedup display.

2. **Metrics honesty stance:** today 0/all-34 sites have confirmed metrics. Do we (a) launch the impact view showing "awaiting confirmation" as the argument, or (b) hold the impact view until ≥N sites confirm at least one sourced metric? Affects whether Phase 4 ships empty-but-honest or waits.

3. **Publish gate strictness:** the spec says "staff stage, never publish," but the workbench is live-write today. Do we want a full staging shadow (heavier build) or a per-section `published` toggle the org flips (lighter, ships in Phase 3)?

4. **Co-design gate:** the org-profile spec is explicit that "nothing ships until an anchor community has shaped it" (Oonchiumpa co-design round). Does the per-site page (Phase 2A) need that co-design round before launch, or can it ship as a read-only stub (no editing) ahead of co-design?

5. **Catch-up scheduling depth:** record-only (`last_interaction`/`meeting_frequency` on existing tables) for Phase 6, or integrate a real calendar (Google Calendar)? Calendar is a larger surface and an external-system write.

6. **Government view access:** is the funder/government view fully public (`?view=government` open to all), or gated behind a login/role? Affects whether funding aggregates and "what scaling would save" figures are public by default.
